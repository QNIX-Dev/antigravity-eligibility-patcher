## ✨ Возможности

- 👥 **Менеджер аккаунтов:** Сохранение, переключение и управление несколькими профилями авторизации (отдельно для CLI/Manager и IDE) без повторного входа через браузер и без аннулирования токенов на сервере.
- 🔓 **Обход ограничений доступности:** Отключение локальных заглушек региональных ограничений ("not available in your location") во всех трех приложениях (CLI, Manager, IDE).
- 🎨 **Интерактивный TUI:** Красивое консольное меню для управления патчами и аккаунтами на базе библиотек `rich` и `questionary`.
- ⚡ **Ядро без зависимостей:** Базовые скриптовые команды работают на стандартной библиотеке Python без необходимости установки сторонних пакетов.
- 🛡️ **Безопасность и откат:** Автоматическое резервное копирование (`*.agybak`) файлов перед модификацией для безопасного отката изменений в один клик.
- ⚙️ **Автоматический поиск:** Умное сканирование реестра Windows, переменных окружения, системного PATH и путей Scoop для динамического обнаружения установленных приложений.
- 🧬 **Устойчивость к обновлениям:** Ищет уникальные сигнатуры и инструкции через регулярные выражения, а не привязывается к статическим файловым смещениям.

## 🚀 Быстрый старт

### Вариант А: Интерактивное меню (рекомендуется)

Для запуска интерактивного TUI-интерфейса с таблицей статусов:

1. **Установите зависимости:**
   ```bash
   pip install -r requirements.txt
   ```
2. **Запустите патчер:**
   ```bash
   python patch.py
   ```

*(Поддерживает навигацию стрелочками, выбор пробелом и подтверждение по Enter).*

---

### Вариант Б: Консольные команды (без зависимостей)

Работает полностью на стандартной библиотеке Python (установка сторонних пакетов не требуется).

| Команда | Описание |
| :--- | :--- |
| `python patch.py status` | Проверить статус патча для всех приложений. |
| `python patch.py patch` | Пропатчить все найденные приложения (резервные копии создаются автоматически). |
| `python patch.py restore` | Восстановить все приложения из бэкапов. |
| `python patch.py patch ide` | Пропатчить только указанные приложения (например, `ide`, `manager` или `cli`). |

> [!TIP]
> Если ваше приложение установлено в нестандартную директорию, автопоиск можно переопределить ручным указанием пути:
> ```bash
> python patch.py --path-cli "D:\CustomTools\agy.exe" patch cli
> ```

## 👥 Менеджер аккаунтов (Поддержка нескольких учетных записей)

Инструмент позволяет сохранять и переключать профили авторизации Antigravity без повторного входа через браузер. Управление разделено на две независимые области:
1. **CLI + Manager** (используют общий токен в Windows Credential Manager).
2. **IDE** (использует собственные ключи авторизации в базе данных `state.vscdb`).

Это позволяет независимо переключать профили для разных приложений и избегать конфликтов блокировки файлов. Профили безопасно сохраняются в Windows Credential Manager.

### Использование в интерактивном меню:
1. Запустите `python patch.py`
2. Выберите пункт **Manage accounts**
3. Выберите целевую область для управления: **CLI + Manager** или **IDE**
4. Вам будут доступны операции сохранения, переключения, удаления профилей или локального выхода.

### Использование в командной строке:
Команды строятся по шаблону: `python patch.py accounts <cli-manager|ide> <действие> [имя]`

| Действие | Пример команды | Описание |
| :--- | :--- | :--- |
| `list` (или `ls`) | `python patch.py accounts cli-manager list` | Список сохраненных профилей для выбранной области с отметкой активного. |
| `save <имя>` | `python patch.py accounts cli-manager save work` | Сохранить текущую активную сессию под указанным именем. |
| `use <имя>` (или `switch`) | `python patch.py accounts cli-manager use personal` | Переключиться на сохраненный профиль. |
| `logout` | `python patch.py accounts cli-manager logout` | Выйти из текущей сессии локально (чтобы войти в другой аккаунт). |
| `rm <имя>` | `python patch.py accounts cli-manager rm work` | Удалить сохраненный профиль. |
| `current` (или `who`) | `python patch.py accounts cli-manager current` | Показать имя текущего активного профиля. |

> [!IMPORTANT]
> Перед переключением профиля или локальным выходом обязательно закройте работающие приложения соответствующей области (CLI/Manager или IDE), чтобы избежать ошибок блокировки файлов базы данных и токенов в памяти.

## 🔍 Детали реализации (под капотом)

<details>
<summary>🛠️ <b>CLI (`agy.exe` бинарный патч Go-бинарника)</b></summary>

CLI выводит при запуске косметическую секцию «Eligibility Check». Решение о её показе принимается в функции `handleAuthResult`, которая читает поле `hasValidAuth` (байт по смещению `+8`) из AuthResult, пришедшего **с сервера**.

1. Патчер сканирует бинарник в поисках сигнатуры гейта, уникальной для всего файла, и отказывается работать, если она отсутствует или встречается более одного раза (защита от незнакомых версий).
2. Сигнатура указывает на проверку: `test rax,rax` → `je` (eligible) → `cmp byte ptr [rax+8], 0` → `jne` (eligible). При нулевом `hasValidAuth` выполнение проваливается дальше и печатает «Eligibility check failed».
3. Инструкция `cmp byte ptr [rax+8], 0` перезаписывается на `test rax,rax` (+`NOP`). Так как `rax` здесь заведомо ненулевой, `jne` всегда уходит в «eligible»-ветку.
4. В результате секция ограничения не рендерится ни при каком ответе сервера.
</details>

<details>
<summary>📦 <b>Manager (`language_server.exe` бинарный патч Go-бэкенда)</b></summary>

Менеджер общается с локальным Go-бэкендом `language_server.exe` (по протоколу connect-rpc) — именно бэкенд выносит вердикт о доступности. Важнее другое: при статусе «недоступно» он не сохраняет OAuth-токен, из-за чего приходится заново проходить авторизацию через браузер при каждом запуске.

1. Патчер сканирует бинарник в поисках сигнатуры, уникальной для всего файла, и отказывается работать, если она отсутствует или встречается более одного раза (защита от незнакомых версий).
2. Сигнатура указывает на проверку `hasValidAuth` в функции валидации логина — инструкцию `cmp byte ptr [rax+8], 0`, сразу за которой идёт привязка токена к структуре статуса.
3. Эта проверка перезаписывается на `mov byte ptr [rax+8], 1` + 2×`NOP` (6 байт суммарно; `NOP`'ы гасят ставший ненужным переход).
4. В итоге аккаунт всегда считается валидным: токен привязывается, сохраняется на диск, а косметический экран ограничения больше не появляется.
</details>

<details>
<summary>💻 <b>IDE (`main.js` патч VS Code)</b></summary>

IDE построена на базе VS Code:
1. Скрипт сканирует файл `resources/app/out/main.js` с помощью регулярного выражения.
2. Ищет минифицированный шаблон авторизации:
   `resetIsTierGCPTos\(\),this\.[A-Za-z_\$0-9]+\.isGoogleInternal`
3. Заменяет его на `resetIsTierGCPTos(),true`, чтобы принудительно выдать права внутреннего разработчика.
4. Очищает кэши байт-кода VS Code (`CachedData` и `Code Cache/js`) для моментального применения изменений.
</details>

<details>
<summary>👥 <b>Менеджер аккаунтов (управление сессиями и профилями)</b></summary>

Функционал переключения аккаунтов работает без обращения к серверам авторизации, чтобы избежать инвалидации токенов (при выходе из приложения стандартным путем сервер аннулирует refresh token).

1. **Разделение хранилищ:** Токены CLI и Manager хранятся в Windows Credential Manager под именем `gemini:antigravity`. Токены IDE хранятся в SQLite базе данных VS Code `state.vscdb` (ключи `antigravityUnifiedStateSync.*`).
2. **Безопасное хранение:** При сохранении профиля (`save`) данные вычитываются из соответствующих хранилищ, кодируются и записываются обратно в Credential Manager под уникальным именем с префиксом `agy-manager:account:cli-manager:<name>` или `agy-manager:account:ide:<name>`.
3. **Обход лимитов размера:** Поскольку Credential Manager накладывает ограничение на размер blob-данных (до 2560 байт), а JSON-сессия IDE (особенно ключ `userStatus`) превышает 8 КБ, данные профиля автоматически разбиваются (шардируются) на части по 2000 байт и сохраняются в виде пронумерованных записей (`.../<index>`).
4. **Синхронизация и защита от перезаписи:** При переключении аккаунтов (`use`) сначала производится проверка блокировки файлов. Если соответствующие приложения запущены, операция блокируется (чтобы избежать потери токенов из-за кэширования в памяти процесса). Перед применением нового токена текущий активный профиль автоматически синхронизируется/сохраняется, сохраняя любые ротированные ключи сессии.
</details>

## ⚠️ Важные примечания

- **Сброс при обновлениях:** При обновлении приложений пропатченные файлы будут перезаписаны оригинальными. Просто запустите `python patch.py patch` заново.
- **Блокировка файлов:** Убедитесь, что патчируемое приложение полностью закрыто перед запуском скрипта, иначе операционная система заблокирует запись в файл.
- **Условия использования:** Изменение проприетарного кода приложений может нарушать условия использования (ToS). Данный проект создан исключительно в образовательных целях — используйте его под свою ответственность.
